## Findings

### [P-1] Three.js eager-bundled into main chunk â€” 291 KB gz extra on first paint
**Where:** `src/pages-public/LandingPage.tsx:2-4`, `src/components/magic/NeuralNetwork3D.tsx:2-4`, `src/components/magic/RobotHologram.tsx:2-4` (imported eagerly by `RoboticsEdgePage.tsx:21` and `FleetPage.tsx:29`).
**What:** `DigitalTwinPage` and `ShowcasePage` are correctly `lazy()`'d in `AppRouter.tsx`, but `LandingPage` directly imports `@react-three/fiber`, `@react-three/drei`, and `three` at module scope. `FleetPage` and `RoboticsEdgePage` do the same indirectly via the two "magic" components. Because the landing page is the default route and Fleet/Robotics are popular nav targets, the 1.05 MB / 291 KB-gz `extends-*.js` chunk is forced into the eager dependency graph alongside main.
**Why it matters:** Stakeholder first impression is page weight. Lifting the 3D out of the critical path is the single highest-leverage change.
**Fix:** `lazy()` the three magic components (`<Suspense fallback={...}>` boundary inside each parent) and `lazy()` the `LandingPage` itself in `AppRouter.tsx`. Expected: main chunk drops ~30-40 KB gz; the heavy Three chunk loads only when 3D is actually rendered.

### [P-2] No route-change motion â€” feels like a tab swap, not a product
**Where:** `src/lib/router.ts:44-48` (`navigate()` mutates `window.location.hash` directly).
**What:** Route changes are instant. With dark glass cards and the cinematic aesthetic, an instant swap reads as "static screens linked by a menu" instead of "one continuous OS surface".
**Why it matters:** This is the cheapest cinematography upgrade available â€” native View Transitions API is supported in all evergreen Chromium (which is what the stakeholder demo will run on), gracefully no-ops elsewhere, zero dep weight.
**Fix:** Wrap `navigate()` in `document.startViewTransition` (feature-detected). Add `::view-transition-old/new(root)` keyframes in `index.css` for a 220 ms crossfade-with-slight-lift.

### [P-3] KPI numbers appear "printed", not "counted"
**Where:** `src/home/KpiStrip.tsx:45-47`, all `src/home/snapshots/*.tsx`.
**What:** The big numeric KPIs (3,247 ops, 184k polys, 98% compliance) render instantly on mount. The hero section already has `framer-motion` slide-up reveals, but the numbers inside don't count up â€” the eye doesn't get to track value.
**Why it matters:** Count-up reveal is the universal stakeholder-dashboard motif. It signals "this is live, this just updated" without needing a fake polling indicator. Bloomberg / Stripe / Linear all use it.
**Fix:** Add a tiny dependency-free `useCountUp(value, durationMs)` hook in `src/lib/hooks/useCountUp.ts`, apply to `KpiStrip` numeric values. Parses `"3,247"` / `"98%"` / `"+12.4"` formats. Respects `prefers-reduced-motion`.

### [P-4] AR line-height is 1.4-ish in places; no `prefers-reduced-motion` honored
**Where:** `src/index.css` global body (no AR-specific rule); animations in `index.css:206-217` (`animate-pulse-ring`, `animate-radar`, `animate-blink`, `animate-drift`) run unconditionally.
**What:** (a) Arabic at 14 px with default ~1.3 leading clips diacritics and reads "compressed" vs the English equivalent â€” Najdi/Arabic typography convention is 1.7 minimum for body. The `rtl-typography-linter` skill flags this as the #1 bilingual-content issue. (b) The radar sweeps, pulse rings, and blinks are visually delightful but **never** stop for users who set `prefers-reduced-motion: reduce` â€” this is an a11y violation that also tanks Lighthouse a11y score (which stakeholders screenshot).
**Why it matters:** Stakeholder pulled "enterprise-grade" claim in the README. Enterprise compliance means respecting `prefers-reduced-motion`. The AR line-height fix is a 2-line CSS edit that lifts every Arabic paragraph on the site.
**Fix:** Add `html[dir="rtl"] body, body { line-height: 1.7 }` for AR-default body text; add `@media (prefers-reduced-motion: reduce) { ... }` block that neutralizes the keyframe-driven animations and disables view transitions.

### [P-5] Magnetic CTA hover absent; KPI tile shimmer is a missed signature
**Where:** `src/home/KpiStrip.tsx:25` (current `hover:-translate-y-0.5` only), `src/index.css:196-203` (`@keyframes shimmer` defined but never applied to a class).
**What:** The shimmer keyframe was already authored but lacks a utility class. KPI buttons get a 2-px lift on hover but no magnetic pull-to-cursor â€” small detail but reads as "premium" when applied. Detail-page CTAs (the `Download PDF`-style action buttons on `_PageShell`/`BrandPage`) get nothing.
**Why it matters:** These are 2 small, high-signal flourishes. The shimmer-on-hover on KPI tiles tells the eye "this is a live data tile, not a static graphic", and magnetic CTAs match the Linear/Vercel motion vocabulary stakeholders associate with 2026 SaaS.
**Fix:** (a) Add `.shimmer-hover` utility in `index.css` that runs the existing `shimmer` keyframe on hover. Apply to `KpiStrip` buttons. (b) Add `useMagneticHover` hook (dependency-free, transform-only) and apply to the actions slot in `_PageShell`.
